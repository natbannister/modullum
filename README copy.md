## Versioning Convention

| Component | Convention | Example
|---|---|---
| Requirements scripts | R(equirements)-[Architecture][Increment] | R-A1
| Code generation scripts | C(ode)-[Architecture][Increment] | C-B11
| Architecture letter | Changes on structural revision | A → B
| Increment | Bumps after any change post clean run | 1 → 2

**Rule:** Once a script runs without programmatic errors, it must be incremented before any further changes *after test artefacts are generated*. Every numbered version is syntactically sound.

## Orchestrator — Agent Head

Sits above all skill modules. Responsible for:

- **Pre-classification** — determines task type before handing off to any skill
- **Pre-flight knowledge check** — queries KB before committing to a run; prevents wasted generation cycles
- **Complexity estimation** — scopes the task, sets clarification budget
- **Skill routing** — selects which skill module to invoke
- **Post-skill planning** — receives structured output, decides next step, decomposition if needed

The agent head provides lightweight framing context to the requirements generator before the interview starts, then resumes control after requirements are output.

### Notes on Schema

- **UID convention:** `R-A1-REQ-004` ties requirement to the script version that generated it
- **Versioning:** Revised requirements increment `version` field — never overwrite. Preserves audit trail without proliferating serial numbers
- **Grouping:** Build flat list during clarification loop, group as post-processing step before handoff. More reliable at 7B scale than maintaining hierarchy live
- **Inter-group dependencies** preferred over individual requirement dependencies — sufficient for most cases without managing a dense graph
- **Dual output:** JSON to KB and downstream nodes; plain text summary to artefacts folder for human readability

### Requirement Groups (Natural Hierarchy)

```
TASK
├── GROUP: Interface

├── GROUP: Functional  
│   ├── Algorithm / method
│   └── Sub-equations (children of parent requirement)
├── GROUP: Validation
│   ├── Input validation rules
│   └── Boundary conditions
└── GROUP: Invariants
    └── Properties that must always hold (e.g. population conservation)
    └── Concrete numerical test cases
```

## Convergence / Pressuriser System (Future)

Without convergence pressure, models ask questions indefinitely. Mechanisms to implement (in order of complexity):

1. **Turn budget with countdown** — model is told N turns remain, prioritises highest-value questions (simplest, implement first)
2. **Forced assumption declaration** — model declares "I will assume X unless corrected" rather than asking; bias toward reasonable assumptions
3. **Confidence scoring** — structured `{"ready": 0.4, "gaps": [...]}` output per turn; readiness-check node triggers handoff at threshold
4. **Hard stop with partial requirements** — after N turns, generate with `[ASSUMED]` flags; downstream test loop becomes ultimate pressuriser

**Key insight:** Clarification loop doesn't need to be perfect — good enough that code generation loop handles the rest. Two pressuriser systems working in series.

## Knowledge Base Layers

| Layer | Technology | Purpose |
|---|---|---|
| Vector store | ChromaDB | Semantic retrieval — docs, notes, code. nomic-embed-text embeddings |
| Structured store | SQLite | Precise fact lookup — task state, entities, tags, relationships |
| Scratchpad | File / in-memory dict | Working memory for multi-step tasks. Intermediate results |
| Workspace manifest | Extension of RunDirectories | Task-scoped run directories, orchestrator-readable manifest |

**Retrieval pattern:** Hybrid — semantic similarity (ChromaDB) for "what's like this?" and tag/key lookup (SQLite) for "what's about this?"

## Conversation Management Pattern

```python
class Conversation:
    def __init__(self, system_prompt):
        self.system = system_prompt
        self.history = []
    
    def add_user(self, content):
        self.history.append({"role": "user", "content": content})
    
    def add_assistant(self, content):
        self.history.append({"role": "assistant", "content": content})
    
    def messages(self):
        return [{"role": "system", "content": self.system}] + self.history
    
    def last_response(self):
        return self.history[-1]["content"]
```

**Key principle:** The model has no memory between calls. The conversation list passed on every call *is* the memory. If history isn't included, the model has no context for prior turns.

**Practical implications:**
- Context window is finite — long histories eat into response token budget
- Every node that needs history must receive it explicitly — nothing shared automatically
- A saved conversation JSON is a resumable state — reload and continue any run from any point
